# Обогащение карточки кейса из исходных волн пользователя

Получаешь:
1. **краткое описание кейса** (его текущая карточка),
2. **сырые фрагменты из 4 волн анализа пользователя** — то, что он сам писал
   про эту организацию/кейс осенью-зимой 2025–2026,
3. **список уже заполненных секций** — их **НЕ ТРОГАЕМ**, не перезаписываем.

Твоя задача — разложить фрагменты по **18 секциям канваса**, сохраняя:
- терминологию пользователя (типы A–F, гипотезы H1–H5, шкалу агентности 0–6,
  фасеты LIN/MOD/NET/META/SWARM и т.д.),
- его аналитическую рамку (ролевая модель ТюмГУ, эволюционные тренды,
  противоречия),
- ссылки на другие кейсы и теорию через [[wikilinks]].

## 18 секций канваса

0. ontology_status — внедрение/прототип/policy/контр-сигнал/симуляция
1. signature_context — где, когда, в какой институции, в каком окне
2. problem_situation — что не работало без AI
3. effect_hypothesis — что обещано
4. ai_architecture — модели, стек, версии, провайдер, RAG, persistent state
5. team_roles — человеческие роли, разнесённые
6. ai_role — функциональная позиция AI в процессе
7. interaction_scenario — пошагово как идёт практика
8. institutional_loop — governance, нормативка, политика данных
9. transit_to_life — pilot → rollout, какие коррекции произошли
10. metrics_evidence — что реально замеряется vs что заявлено
11. risks — goal-substitution, lowered bar, weak audit trail, vendor lock-in
12. countersignals — где практика идёт против заявленного
13. transferability — куда этот паттерн можно перенести
14. theory_links — какие типы / гипотезы / противоречия / моды затронуты
15. open_questions — что осталось неясным
16. next_wave_followup — что обязательно перепроверить в следующем дифф-прогоне
17. sources_verification — что подтверждено, что под вопросом

## Правила

- **Не выдумывай.** Если про конкретную секцию во фрагментах нет данных —
  оставь её пустой (не возвращай вообще). Лучше пустая секция, чем галлюцинация.
- **Не переписывай уже заполненные.** Их id даны в списке `existing_sections` —
  пропусти их.
- Используй формулировки пользователя там, где они есть. Не сглаживай их в
  «правильный академический язык» — пользователь нарочно так пишет.
- Каждая заполненная секция — 2–5 абзацев, по делу. Не вода.
- Связи с теорией пиши через [[wikilinks]]: [[H4-...]], [[A-governed-access]],
  [[autonomy-vs-control]], [[acceleration]] и т.п.
- Если в фрагментах есть цифры (пользователи, заявки, проценты) — обязательно
  включи их в metrics_evidence.

## Возврат: строгий JSON

```
{
  "sections": {
    "<section_id>": {"text": "<2-5 абзацев md>", "confidence": "low|medium|high"},
    ...
  },
  "skipped_sections": ["<id секций, которые остались пустыми из-за нехватки данных>"],
  "notes": "<2-3 предложения: что было особенно интересного в этих фрагментах, какие связи всплыли>"
}
```

Запреты:
- никаких заголовков `##` внутри text — только абзацы
- никаких упоминаний «я», «as an AI», «based on the provided fragments»
- ничего не сочиняй — если данных мало, пиши коротко и честно
- не дублируй existing_sections даже если в фрагментах есть про них больше
