## A. Куда движется продукт

Сейчас Paideia это:

1. **Корпус** — 110 кейсов + 6 типов + 5 гипотез + 7 противоречий + 5 мод + 6 контр-сигналов; всё в md+git, индексировано в SQLite.
2. **Канвас 18 секций** на каждый кейс; LLM-агент с веб-поиском заполняет пустые секции.
3. **Поиск/каталог/карта/подбор аналогов/мониторинг волн/CRUD теории** — есть, через FastAPI + Jinja.
4. **5 базовых юскейсов LLM-агента**: section-filler, url-importer, wave-runner, hypothesis-rechecker, match-explainer.

Следующая логическая ступень — превратить базу из **read-write архива** в **рабочий инструмент проектировщика образовательных программ**, который не только показывает кейсы, но и:
- помогает **проектировать собственное вмешательство** с использованием теории корпуса;
- **обстреливает проект** разными позициями и сценариями;
- **симулирует** эффекты и собирает выходные документы.

Это вторая половина «недельного» уровня из первоначального плана.

## C. Каталог пользовательских сценариев

Условные обозначения:
- **🧠** — концептуальный сценарий (помогает думать)
- **🔧** — инженерный сценарий (даёт артефакт)
- **🎭** — позиционный/ролевой (через стейкхолдеров)
- **🌱** — генеративный (предлагает решения)
- **🧪** — стресс-тестовый

Каждый сценарий описан как:
**ИМЯ** · *коротко зачем* · вход → выход · стек/промпт · 0.5–1 вечер на реализацию.

### Семья 1 · Перенос и синтез _(минимум, есть/в работе)_

1. **Поиск аналогов** ✅ ·
   /match: текст или оси → топ кейсов.

2. **Перенос свойств** _(новый)_ · 🌱
   Берёшь кейс X и переносишь его архитектуру на свой контекст: «возьми Oxford-D-контур governance и адаптируй под региональный российский вуз с 1.5k студентов».
   *Вход:* case_id + project_id. *Выход:* draft канваса проекта с подсвеченными секциями, заполненными по аналогии + список «что не переносится и почему».
   *Промпт:* `transfer.md`, gpt-5 (нужны рассуждения).

3. **Анализ проекта через корпус + теорию + RAG-документы** _(новый)_ · 🧠
   Полное «осмысление» проекта: его место на карте, ближайшие типы, какие гипотезы он подтверждает/опровергает, какие противоречия в него встроены, какие из твоих документов на это указывали.
   *Вход:* project_id. *Выход:* отчёт 4–8 страниц с цитатами из корпуса и из твоих документов.
   *Промпт:* `project_synthesis.md`, gpt-5 + sonar для пропущенных кусков.

### Семья 2 · ТРИЗ для педдизайна

4. **Поиск противоречий внутри проекта** · 🧠
   Что мешает чему: «глубина погружения мешает массовости», «индивидуальная траектория мешает оценочной согласованности».
   *Вход:* канвас проекта. *Выход:* список пар «требование А ↔ требование Б», размеченных как административные/технические/педагогические.
   *Промпт:* `triz_contradictions.md`, gpt-5.

5. **ИКР (идеальный конечный результат)** · 🧠
   Для каждой ключевой функции — формулировка «функция выполняется сама, без затрат ресурса». Это специальная провокация, не план.
   *Вход:* функция или весь проект. *Выход:* набор ИКР-формулировок + указание на ближайшие кейсы корпуса, в которых ИКР отчасти реализован.
   *Промпт:* `triz_ikr.md`, gpt-5.

6. **40 приёмов ТРИЗ под педдизайн** · 🌱
   Каталог приёмов разрешения противоречий, переинтерпретирован: разъединение → split-курс, динамизация → адаптивная траектория, посредник → AI-роль, обратная связь → автогрейдинг, локальное качество → персонализация рубрик, и т.д.
   *Вход:* противоречие из (4). *Выход:* топ-5 применимых приёмов + примеры из корпуса.
   *Промпт:* `triz_40methods.md` + справочник `triz_methods_edu.yaml`.

7. **Веполь-анализ** · 🧠
   Триада «студент — содержание — посредник». Где в проекте посредник пустой? Где он перегружен? Где AI занимает место, где должен быть человек?
   *Вход:* канвас. *Выход:* схема веполя + предложения по балансировке.

8. **Эволюционные тренды** · 🌱
   8 линий эволюции технических систем, применённых к образовательным программам (моно→би→поли; макро→микро; жёсткое→гибкое; человек→машина). Куда твой проект уже сдвинулся, куда не сдвинулся, что выходит из моды.
   *Вход:* канвас + первое внедрение. *Выход:* профиль зрелости по 8 линиям.

### Семья 3 · Стейкхолдер-обстрел (Кайроскоп)

> Кайроскоп здесь — мне нужна твоя версия определения. Я предполагаю: «удерживание разных временных горизонтов и моментов действия разных позиций». Если другое — поправь.

9. **Карта стейкхолдеров проекта** · 🎭
   Авто-генерация по канвасу: студенты, преподаватели, ректор, методическое управление, IT, юрист, родители, региональный регулятор, бизнес-партнёр, общественный наблюдатель — кто релевантен. По каждому: интересы, страхи, рычаги, временной горизонт.
   *Вход:* канвас проекта. *Выход:* доска стейкхолдеров (md+граф).

10. **Атака от лица каждого стейкхолдера** · 🎭🧪
    Для каждой позиции — 5 самых неприятных вопросов к проекту. Для министра: ФГОС? трудоустройство? стоимость? риски репутации? для студента: «зачем мне это, если зачёт всё равно сдаём вы»? и т.д.
    *Вход:* проект + карта стейкхолдеров. *Выход:* лист вопросов + предложенные ответы, помеченные как «выдерживает / не выдерживает».
    *Промпт:* `stakeholder_attack.md`, gpt-5 + role-playing.

11. **Социальное трение / сопротивление** · 🧪
    Где в твоём проекте конфликт идентичностей? Кто чувствует угрозу профессии? Где появляется «AI = списывание»? Где возникает страх «меня заменят»?
    *Вход:* канвас + карта стейкхолдеров. *Выход:* карта трения с прогнозом интенсивности 1–5.

12. **Проверка на нормативку и этику** · 🧪🎭
    ФГОС, ФЗ об образовании, обработка ПДн, этический кодекс. Где проект может застрять? Подсказывает релевантные нормативные акты + кейсы корпуса с похожими столкновениями (e.g. Reading, Sydney).
    *Вход:* канвас. *Выход:* список нормативных рисков + цитаты из документов.

### Семья 4 · Симуляция и стресс-тест (Литопс)

> Литопс — про что? Я допускаю: «типология устойчивых форм, способных пережить экстремальные условия» (от южноафриканского растения). Поправь.

13. **Симулятор первого семестра** · 🧪
    Прогон проекта по неделям: неделя 1 — старт, неделя 5 — первое падение мотивации, неделя 8 — конфликт с группой, неделя 12 — экзаменационный кризис. На каждой неделе — что произойдёт, что отвалится, что сломается.
    *Вход:* канвас + параметры (длина курса, число студентов, уровень). *Выход:* week-by-week прогноз + декомпозиция рисков.
    *Промпт:* `semester_sim.md`, gpt-5.

14. **Стресс-тесты «что если»** · 🧪
    Серия what-if: «бюджет −50%», «API закрыли», «ключевой преподаватель ушёл», «регулятор требует прозрачности», «студенты массово отказались от AI». Что ломается первым? Что выживает?
    *Вход:* канвас. *Выход:* матрица «сценарий × секция канваса × выдерживает/нет».

15. **Симулятор выпуска** · 🧪🧠
    Что реально приобретут студенты к концу программы? Какие компетенции декларируются, какие формируются, какие иллюзорны? Какие — «знание о», а какие — «способность делать».
    *Вход:* канвас. *Выход:* список заявленных vs реалистично формируемых компетенций.

16. **Anti-agent-washing аудит** · 🧪
    Проверка: твой проект agentic или это «чат-бот в обёртке»? Сверка с критериями из корпуса (autonomy / persistent state / multi-step / audit trail). Возвращает заявленный и фактический уровень агентности.
    *Вход:* канвас. *Выход:* агентность 0–6 с обоснованием каждого балла.

## D. Что ещё за рамками этих 26, что предлагаю обсудить

## F. Library v2+ — отбор и голос

### F1 · Консистентная выборка из противоречащих источников
Книги в библиотеке часто противоречат друг другу (бихевиоризм vs Выготский, инструментализм vs критическая педагогика, агентность vs governance). При сборке контекста для разговора об ИИ нужен **selector**, который:
- видит группы концептов-антагонистов (через embedding-кластеризацию + LLM-разметку «conflicts_with»);
- для конкретного запроса выбирает **одну** консистентную ветку (или явно сообщает «есть две школы: X и Y, какую берём?»);
- помечает в RAG-контексте «эта цитата — из позиции А, эта — из контр-позиции Б»;
- умеет режим «steelman» — собрать максимально сильную версию одной из сторон, не смешивая с противоположной.

Это разворот текущего hierarchical-RAG из «найти всё похожее» в «найти согласованное».

### F8 · Mobile / touch UI

Большинство пользователей будут с телефона, не с десктопа. Сейчас все интерфейсы (Cmd+K, dropdown-навигация на :hover, многоколоночные канвасы, sticky-кнопки 80px) спроектированы под mouse + клавиатуру.

**Что нужно:**
- Responsive breakpoints (`sm:` Tailwind = 640px, `md:` = 768px) — пройти по всем шаблонам: catalog, project workspace, canvas, library, theory, map
- Touch-friendly navigation: гамбургер-меню вместо `:hover`-dropdown; bottom-tab-bar для главных разделов (Корпус / Проекты / Библиотека / Концьерж)
- Канвас 18 секций → swipeable stack или accordion на mobile (не grid)
- Карта корпуса (`/map`) — pinch-zoom, tap-to-detail, не click-to-detail
- Cmd+K → плавающая кнопка-FAB «🔍 спросить» снизу справа
- Формы (wizard, upload) — крупные input + viewport meta + не зум при focus
- Concierge ответы — карточки suggested_actions в одну колонку, не grid
- Тёмная тема (опционально, но почти бесплатно через Tailwind `dark:`)

**Не нужно:** отдельное PWA / нативное приложение. Mobile web достаточно.

### F9 · Onboarding + role assignment + product tour

Первая встреча с продуктом сейчас — пустая landing страница с «спроси что угодно». Юзер не понимает: «что это вообще?», «кто я тут?», «с чего начать?».

**Что нужно:**
- **Onboarding screen** (3-4 экрана / 30 сек): «что это», «что внутри» (118 кейсов + теория + библиотека + проектный движок), «как с этим работать»
- **Role assignment** — на старте выбор: «я преподаватель → проектирую курс», «я методолог → исследую корпус», «я администратор/декан → ищу аналоги для программы», «я студент → разбираю материалы». От роли зависит default-сценарий и видимость разделов
- **Scenario binding** — после выбора роли предложить 2-3 «первых сценария» для этой роли (Course Wizard для преподавателя, Match-search для методолога и т.д.)
- **Tour-overlay** — coachmarks для ключевых элементов (концьерж, библиотека, проекты, Cmd+K)
- **Skip + redo** — onboarding в любой момент через `/onboarding`, для возврата к выбору роли

### F10 · Модель-селектор + переключение на дорогую при сложных вопросах

Сейчас концьерж всегда использует `chat_fast` (gpt-4.1-mini). На сложных вопросах (методологический разбор, противоречия, многошаговый анализ) этого не хватает.

**Что нужно:**
- Кнопка переключения модели в чате: 🚀 fast (default) / 🧠 deep (gpt-5) / 🌐 search (sonar-pro)
- Auto-suggestion: если концьерж detected `intent` = `compare_two` / `theory_question` / `dropped_project` со сложной структурой → в ответе предложить «попробуй на deep-модели — это сложнее, чем я могу быстро» с кнопкой re-ask
- Метка стоимости запроса: «≈ $0.002» рядом с кнопкой, чтобы юзер понимал
- Telemetry: считать что % запросов реально переключают на deep — показатель «слишком тупая default»

### F11 · Rate limits + квоты разговоров

API-ключ один на всех (`LLM_PRIMARY_API_KEY` Сашин). Без лимитов один активный юзер может за вечер сжечь $50-100.

**Что нужно:**
- Per-IP rate limit на `/api/ask` и сценарии: например 30 запросов/день fast, 5/день deep, 3/день search
- Глобальный daily budget cap: при превышении $X в день — концьерж отвечает «дневной лимит платформы исчерпан, попробуй завтра»
- Soft-warning юзеру: «осталось 5 запросов сегодня»
- Сохранять daily spend в `llm_runs` (там уже есть tokens_total), показывать в `/llm-runs` дневную/месячную сводку
- Опционально: персональные ключи через `/settings` — кто хочет без лимитов, приносит свой 302.ai ключ

### F12 · Касса для пожертвований

Если платформа открыта для внешних юзеров и сжигает деньги на LLM — нужен механизм компенсации.

**Что нужно:**
- Страница `/support` с описанием себестоимости («один глубокий разбор ≈ $0.05; платформа в месяц ≈ $X»)
- Кнопки: ЮMoney / Тинькофф / Boosty / Patreon (для зарубежных) — без сложной OAuth-интеграции, просто ссылки на платёжные страницы
- Опциональный thank-you модал после 10-го запроса: «нравится? поддержи»
- Прозрачность: рядом с балансом показывать «получено за месяц / потрачено на LLM» — публичный счётчик
- Если пользователь донатил → его IP/cookie помечен `supporter` и снимаются rate limits

### F3 · Course as first-class entity + Participant interface

**Концептуально.** Сейчас Paideia — исследовательский инструмент (корпус + теория + проектировщик). Эта ветка добавляет **операционную** ось: курс как живая последовательность событий, где пользователь не только автор замысла, но и участник с ролью.

**Три онтологических статуса текста:**
- `library_book` — канонический теоретический текст (Выготский), концепты-инварианты
- `corpus_case` — наблюдаемая практика других (118 кейсов)
- `course_artifact` — событие в моей собственной практике (лекция, чат, домашка) — обрабатывается **litops-логикой**: вопрос → модель → задача → реакция, а НЕ как источник знания

**Новые сущности:**
```
Course (id, название, период, авторы, роли, default_for_user?)
  └── Event[] (lecture / seminar / homework_release / homework_submission /
                chat_burst / reading / exam / reflection_session)
        ├── timestamp, duration, artifacts[] (транскрипт, слайды, аудио, тексты студентов)
        └── litops_extracts[] (вопросы аудитории, предъявленные модели,
                                затруднения, инсайты, открытые петли)
```
`litops_extracts` — первоклассная сущность отдельно от концептов: концепт = «что есть в теории», extract = «что случилось в этой комнате с этими людьми в этот момент».

**Три контекстных режима концьержа** (расширение F1):
1. Вне курса — корпус + теория + библиотека (как сейчас)
2. Внутри курса как автор — + полная история событий, чтобы не противоречить себе
3. Внутри курса как участник (student/seminarist) — фильтр по `timestamp ≤ now` (не спойлерить будущее) + видимость по роли

**Куда садятся существующие куски:**
- Library v1 — тот же ingest, новые типы экстрактов
- Wizard (6 типов преподавателя) → фабрика Course-ролей
- Stakeholders (7 атак-профилей) → «персонажи курса» для симуляции аудитории
- Concierge → получает обязательный `course_context` (None | course_id+role)
- F2 voice-agent → главный интерфейс семинариста («методолог в ухе» во время живой пары)

**Фазы:**
| # | Что | Слож. |
|---|-----|-------|
| G1 | Course-схема (courses, course_events, course_role_bindings, litops_extracts) + миграция | 1 в |
| G2 | Course Workspace `/courses` + `/course/{id}` с лентой событий, ручное добавление события | 1-2 в |
| G3 | Litops Digestor — транскрипты (.vtt/.srt/.txt) → автоэкстракция (вопросы / модели / непонятые места / открытые петли). Промпт `litops_extract.md` | 2 в |
| G4 | Participant Lens — переключатель роли в шапке; концьерж фильтрует RAG по timestamp+role | 1-2 в |
| G5 | Импорт чатов (Telegram/Discord/Zoom JSON) → события `chat_burst` с кластеризацией | 1-2 в |
| G6 | Reflection Session — «методологический протокол курса»: что сработало / зависло / непереварено | 2 в |
| G7 | Multi-course mode — переключатель «в каком курсе мы» в навигации, `default_course` в профиле | 1 в |

Порядок: **G1→G2→G3→G4** = MVP участника. G5→G6 углубление. G7 раньше, если используешь платформу одновременно для разных курсов.

**Развилки для решения:**
1. Канонизация артефактов: после semester end может ли `course_artifact` промотироваться в `library_book`?
2. Privacy студенческих материалов: анонимизировать перед litops, или скрыть от не-author ролей?
3. Источник timestamp событий: метаданные файла / имя файла / ручной ввод?
4. Связь с `projects`: Course = отдельная сущность или `project.runs[]` (project = замысел программы, course = исполнение)?
5. F2 внутри Course: realtime (live transcript → подсказка через 2 с) или post-hoc разбор?

### F2 · Голосовой движок «педагог-методолог»
На основе library + corpus собрать **voice-agent для участия в живых семинарах и обсуждениях**:
- **роли на выбор**: педагог / методолог / эпистемолог / критический философ / провокатор;
- **режим работы — НЕ отвечать**: задавать вопросы, подводить, спорить, критиковать, выводить в рефлексию методов;
- эпистемически честная позиция: «я знаю, чего не знаю», «это в моей модели → так, в твоей → иначе»;
- умеет вытаскивать **скрытые допущения** оппонента, **проблематизировать** ход, требовать **критериев истинности**;
- голос (TTS) + STT realtime через WebRTC; latency < 1.5 с;
- лог семинара → автоматический «методологический протокол» (что обсудили, какие тупики, какие открытые вопросы).
Требует: F1 (без selector ответы будут эклектичными), realtime LLM (gpt-realtime / sonar-voice / similar), голос (ElevenLabs или 302.ai voice).
